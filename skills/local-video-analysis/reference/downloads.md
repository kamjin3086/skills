# Download Reliability Reference

Use this reference when the input is a URL, a download fails, or a site-specific retry is needed.

## Stable Default

Use the evidence pipeline first; it creates a run directory outside the repo, downloads into its `download` folder, normalizes subtitles, and writes a single manifest:

```bash
python skills/local-video-analysis/scripts/prepare_evidence_pipeline.py "$URL" --label video-url
```

Read `pipeline_manifest.json` and continue only when `ok=true` and `video_path` exists. Use `download_video.py` directly only for manual retries or debugging.

The pipeline installs missing Python helper tools, including `yt-dlp`, into `~/.cache/local-video-analysis/tools/venv` by default. If that install fails, it stops at `download_unavailable` and asks for an isolated install or a local file.

For manual isolated `yt-dlp` installs, prefer one of:

```bash
python -m pipx install yt-dlp
# or inside an explicit venv
python -m venv ~/.cache/local-video-analysis/yt-dlp-venv
~/.cache/local-video-analysis/yt-dlp-venv/bin/python -m pip install -U yt-dlp
PATH="$HOME/.cache/local-video-analysis/yt-dlp-venv/bin:$PATH" \
  python skills/local-video-analysis/scripts/prepare_evidence_pipeline.py "$URL"
```

Avoid `sudo pip install` and global package installs.

## Retry Ladder

Use the cheapest likely fix first:

1. Update `yt-dlp` if the extractor is old, the site changed, or Bilibili returns `HTTP 412`.
2. Retry with browser cookies: `--cookies-browser chrome` or `--cookies-browser firefox`.
3. Lower bandwidth/format pressure: `--max-height 480`.
4. Reduce request pressure for throttling/rate limits: `--sleep-requests 1 --concurrent-fragments 1`.
5. If login, region, member-only, age check, or DRM still blocks the download, ask the user for a local file or valid cookies.

Do not spend a long time inventing extractor flags. After two sensible retries, report the blocker.

## Site Notes

| Site/symptom | Likely cause | Best next step |
|---|---|---|
| YouTube sign-in/age check | cookies needed | Retry with browser cookies |
| YouTube throttling/slow fragments | request pressure | Use `--sleep-requests 1 --concurrent-fragments 1` |
| Bilibili `HTTP 412` | old extractor or anti-bot | Update `yt-dlp`; retry with Chrome cookies |
| Bilibili missing title/video after success | multi-part or unusual page | Check `download_info.json` and sidecars; if needed ask for exact episode/part |
| Region/member/DRM block | access restriction | Ask for local file; do not brute-force |

## Storage And Cleanup

Default run artifacts live under `~/.cache/local-video-analysis/runs/<timestamp>-<label>`, never inside the skill directory. Keep these during analysis:
- downloaded video,
- sidecar subtitles/info JSON,
- extracted WAV,
- compressed frames/contact sheets,
- transcripts and visual JSON.

This is an intentional space-for-time tradeoff: reusing downloaded media, extracted audio, and compressed frames avoids repeating expensive network and ffmpeg work. `prepare_run_dir.py` prunes runs older than 14 days by default; use `--prune-days 0` when the user wants to keep artifacts.

`download_video.py` stores incomplete fragments in `.partial` and removes them after success unless `--keep-partials` is used.

## Keep The Pipeline Fast

- Download at `--max-height 720` by default. Use 1080p only when small text or visual details matter.
- Prefer sidecar subtitles from the platform when available, but still inspect audio if subtitles look autogenerated or incomplete.
- Do not run dense frame extraction across an entire long video first. Start coarse, refine only important segments.
- Avoid repeated full-video `describe` calls after an empty/generic result; switch to segment frame evidence and targeted questions.
